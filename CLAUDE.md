## Принцип работы — Atomic Design

**Источник истины — библиотека компонентов и токены, не страницы.**

При любой правке всегда движемся **от атомов к организмам**:

1. **Изменился цвет/размер/шрифт** → правим **токен** в `styles/tokens.css`. Не меняем значения в компонентах напрямую — только через токены.
2. **Изменилось поведение/раскладка компонента** (кнопка, событие, life-line) → правим в `styles/components/<имя>.css` и его секцию в `components.html` (kitchen sink).
3. **Изменился композит** (Header, Sidebar, Timeflow, Popup, Timeline-mini) → правим CSS компонента и его JS-рендер; страницы автоматически подхватят.
4. **Только в самом конце** — раскатываем изменения на использующие страницы (`index.html`, и т.д.).

**Никогда не правим прямо на странице, минуя библиотеку.** Если на странице добавляется новый класс/правило — оно сначала ложится в библиотеку, потом используется на странице.

**После любых правок в библиотеке — обновить kitchen sink (`components.html`)**, чтобы demo отражали актуальную структуру. Иначе библиотека рассинхронизируется с реальностью.

Когда пользователь говорит **«фиксируем это»** — записываем суть/поведение в соответствующий файл (обычно `MAIN_SCREEN.md` или `BACKLOG.md`).

## Что фиксировать (помогаем пользователю решать)

Пользователь не всегда знает, что должно идти в md, а что — нет. Claude активно решает за него по правилам:

- **`MAIN_SCREEN.md`** — суть и поведение системы / компонентов / композитов; формулы (`x = (year − START) × pxPerYear`); ментальные модели; принципиальные константы (типа «4px gap», «8px от края»); правила взаимодействия (hover, click, scroll).
- **`BACKLOG.md`** — отложенные задачи и долги (то что нужно сделать позже).
- **Комментарий в коде** — тонкие пиксельные правки конкретных значений без принципиального смысла; технические детали реализации (через `::after` vs gradient).
- **Не фиксируем нигде** — одноразовые правки и утверждения «давай сделаем X» если они уже воплощены в коде и не несут принципа.

В конце каждой подсессии в ответе пользователю явно говорить: «фиксирую в MAIN_SCREEN.md → блок X», «не фиксирую — это в комментарий кода», «добавил в BACKLOG.md → дизайн-долг».

## Проект
Интерактивный исторический таймлайн 1850–1950. Цель: показать, что исторические личности жили в одно время и были связаны друг с другом.

## Стек
- Vite + vanilla JS (без фреймворков)
- CSS Custom Properties (токены из Figma)
- JSON-файлы для данных (без сервера)
- Деплой: Netlify

## Структура файлов
```
data/
  people.json     — персонажи (43 записи)
  events.json     — мировые события (12 записей)
  roster.json     — эталонный список: кто запланирован и кто уже добавлен
  _schema.js      — схема данных (источник истины по полям)
styles/
  tokens.css      — дизайн-токены из Figma (генерируется после чистового дизайна)
  main.css
js/
  timeline.js
  filter.js
  popup.js
.claude/commands/
  add-person.md   — скилл /add-person
  add-event.md    — скилл /add-event
  check.md        — скилл /check
```

## Данные — ключевые правила
- Имена и описания на **русском**, `fullName` на языке оригинала
- Фото — только общественное достояние (Wikimedia Commons предпочтительно)
- Временной диапазон событий: **строго 1850–1950**
- Связи между людьми хранятся **у обоих** (двусторонние)
- Перед добавлением человека/события — проверить `roster.json` (поле `added`)
- После добавления — обновить `added: true` в `roster.json`

## Правила экономии токенов при работе с Figma

Пользователь — дизайнер, рисует сам. Моя зона — аудит системы, токены, код. Поэтому:

1. **Каждый фрейм читаю один раз.** Результат сразу складываю в `design-snapshot.md`. Дальше работаю со снапшотом, в Figma не возвращаюсь без необходимости.
2. **Перед `get_design_context` использую `get_metadata`** — дешевле, даёт структуру и имена, часто этого достаточно.
3. **Скриншоты беру только для сверки визуала**, не при каждом чтении.
4. **Не рисую макеты и компоненты в Figma через MCP.** Дизайнер делает это быстрее и дешевле. Моя помощь — текстовый чек-лист "что дорисовать / какие состояния добавить".
5. **Переменные (variables) в Figma писать можно** — это точечная операция, дёшево. Большие правки слоёв — нет.
6. **"Покажи ещё раз" — сначала в снапшот.** Если в снапшоте нет нужной детали, тогда читаю Figma точечно (один фрейм/нод), не всю страницу.
7. **Темы держим раздельно.** Разговор про токены — отдельно от разговора про код. Длинный контекст = дорогие следующие сообщения.
8. **При аудите страниц** иду от metadata всей страницы → список фреймов → читаю только то, что нужно для текущего шага.
