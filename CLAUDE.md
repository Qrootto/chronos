# CLAUDE.md (проект Past Simple)

## Контекст

**Что это:** интерактивный исторический таймлайн 1850–1950. Цель — показать, что исторические личности жили в одно время и были связаны друг с другом.

**Уровень проекта:** 2 — личный проект (см. правило 6.1 глобального CLAUDE.md).

**Глобальные правила:** действуют все, см. `~/.claude/CLAUDE.md`. Этот файл их **дополняет**, не заменяет.

## Старт сессии

При начале сессии прочитать в порядке:
1. `~/.claude/CLAUDE.md` — глобальные правила (загружается Claude Code автоматически).
2. Этот файл (`CLAUDE.md`) — проектные дополнения.
3. `SESSION.md` — где остановились в прошлой сессии.

## Достоверность

Past Simple — это исторический проект. **Вся информация о людях и событиях должна быть достоверной на 100%.** Это ядро доверия пользователя к продукту.

Правила:
- **Не выдумывать факты, даты, имена, цитаты, цифры.** Если точная цифра неизвестна — не писать её, переформулировать общими словами.
- **Не использовать сомнительные/анекдотичные источники** (популярные мифы, апокрифы, «есть мнение»). Только проверяемое.
- **Все тексты, которые я создаю** (events.description, events.sections, любые подписи) — это **черновик до верификации**. Перед production пройтись через сверку с Wikipedia / Britannica / другими надёжными источниками. Сейчас все мои тексты — черновики.
- **Если сомневаюсь — явно говорю об этом пользователю**, не «маскирую» неуверенность красивой формулировкой.
- **При факт-чек-проходе** использовать `WebFetch` на Wikipedia и сверять каждое утверждение. Если не подтверждается — переписать или удалить.

См. также пункт «Факт-чек текстов» в `BACKLOG.md`.

## Данные и домен

Источник правды по схеме данных, диапазону лет и перечислениям (категории людей, типы связей) — `data/_schema.js`. В этом файле не дублируем числа и списки, ссылаемся.

Доменные правила:
- Имена и описания на русском, `fullName` на языке оригинала.
- Фото — только public domain (Wikimedia Commons предпочтительно).
- Связи между людьми двусторонние (хранятся у обоих).
- Перед добавлением человека/события — проверить `roster.json` (поле `added`); после добавления — обновить `added: true`.

## Работа с Figma MCP

**Главный принцип.** Дизайнер рисует сам, Claude помогает с аудитом и токенами. Точечная запись variables в Figma — допустимо. Большие правки слоёв и создание макетов через MCP — нет, дизайнер делает быстрее. Помощь с дизайном — текстовый чек-лист «что дорисовать / какие состояния добавить».

**Workflow чтения:**
- Каждый фрейм читаю один раз, результат сразу в `design-snapshot.md`.
- Перед `get_design_context` — `get_metadata` (дешевле, часто достаточно).
- Скриншоты — только для сверки визуала, не при каждом чтении.
- «Покажи ещё раз» — сначала в снапшот; если детали нет, точечный read одного фрейма/нода, не всей страницы.
- При аудите страниц — от metadata всей страницы → список фреймов → читаю только нужное для текущего шага.

## Известные ловушки

1. **Vite + production CSS-каскад.** В prod-билде Vite может переставлять порядок `<link>`-тегов → нельзя полагаться на порядок файлов в каскаде. После CSS-правок обязательно `npm run build && npm run preview`.
2. **Дублирование констант между документацией и схемой.** Из ретроспективы: диапазон лет 1850–1950 разъехался между CLAUDE.md, MAIN_SCREEN.md и скиллами (последние говорили 1880). Источник правды — `data/_schema.js`. Любое появление чисел в `.md`-файлах — повод заподозрить дрифт.
3. **Stale design-snapshot.** `design-snapshot.md` тихо устаревал (`brand/100 = #5c78db` против реального `--brand: #8856fd` в `tokens.css`). Снапшот — кэш, не источник правды (правило 2.2).

## Проверка при старте сессии

1. `design-snapshot.md` ↔ Figma file `UWWMIMoOJUYRhoEk5byWs4` ↔ `styles/tokens.css` — соответствие токенов (цвета, шрифты).
2. Скиллы `.claude/commands/add-person.md` и `add-event.md` ↔ `data/_schema.js` — соответствие диапазона лет и перечислений.
3. `BACKLOG.md` ↔ `TODO`-комментарии в коде — нет ли долгов в коде, которых нет в беклоге, и наоборот.
