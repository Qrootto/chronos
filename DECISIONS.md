## 2026-05-04 — Старт проекта Chronos prototype

**Контекст:** Артём — продуктовый дизайнер с базовым знанием HTML/CSS, без фреймворкового бэкграунда. Хочет создать интерактивный исторический таймлайн (1850–1950), показывающий, как великие исторические личности жили в одно время и были связаны друг с другом.

**Решение:** Стек — Vite + vanilla JS (без фреймворков). CSS Custom Properties для дизайн-токенов. JSON-файлы для данных (без сервера). Изначальный хостинг — Netlify.

**Альтернативы:** Альтернативы не задокументированы в истории.

**Последствия:** Прототип развился в живой проект. Привёл к серии решений по дизайн-системе (2026-05-05), принципу достоверности (2026-05-04), миграции хостинга (2026-05-04), приватности шрифтов (2026-05-06).

## 2026-05-04 — Принцип достоверности всего исторического контента

**Контекст:** Past Simple — исторический проект. Тексты о людях и событиях создаются с участием Claude (drafting), но любая ошибка подрывает доверие пользователя к продукту, потому что он видит сайт как авторитетный источник.

**Решение:** Зафиксировать как критический принцип проекта: вся информация о людях и событиях должна быть достоверной на 100%. Не выдумывать факты, даты, имена, цитаты. Не использовать сомнительные источники. Все тексты, которые создаёт Claude — черновики до верификации (сверка с Wikipedia / Britannica). Принцип записан в проектный `CLAUDE.md` как раздел «Достоверность».

**Альтернативы:** Альтернативы не задокументированы в истории.

**Последствия:** Все тексты в `public/data/people.json` (181 секций по 43 людям) считаются черновиками до фактчека. Сэмпл-фактчек 2026-05-04 показал ~27% брака, что подтвердило важность принципа. Задача «факт-чек текстов» зафиксирована как приоритет №1 в `BACKLOG.md → К релизу`.

## 2026-05-04 — Переименование Chronos → Past Simple

**Контекст:** Изначальное рабочее имя `Chronos` оказалось недостаточно отличительным для публикации. Перед production-выкаткой нужно публичное имя.

**Решение:** Переименование в `Past Simple` (отсылка к английскому грамматическому времени). Затронуто: HTML titles, бренд в навигации, `package.json` (`past-simple`), глобалка `__pastSimple`, User-Agent, comment-headers, все `.md`-документы.

**Не переименовано (compromise):** GitHub-репозиторий `Qrootto/chronos`, локальная папка `Chronos/`, Netlify site name (`coruscating-entremet-9241f5`), имена memory-файлов с префиксом `chronos_*`.

**Альтернативы:** Альтернативы не задокументированы в истории.

**Последствия:** Внешнее имя везде согласовано. Внутренние имена (репо, папка, memory-файлы) остались под старым именем — это технический долг низкого приоритета (не влияет на продукт, влияет только на удобство ориентирования в инструментах).

## 2026-05-04 — Миграция Netlify → Cloudflare Pages

**Контекст:** Netlify free tier работает в credits-системе: 1 deploy = 15 credits, 300 credits/мес = ~20 деплоев/мес. За день переименования + правок исчерпали 315/300, проект paused до 1 июня. Нужен бесплатный хостинг с большим лимитом деплоев.

**Решение:** Миграция на Cloudflare Pages (бесплатно, 500 builds/мес, unlimited bandwidth). DNS переехал на Cloudflare nameservers (заменили reg.ru). HTTPS auto.

**Альтернативы:** Альтернативы не задокументированы в истории.

**Последствия:** Деплой стал effectively unlimited по объёму. URL `/library` поднят через redirect одной задачей с миграцией (коммит `333d330`). Все упоминания «Netlify» в документации — теперь legacy / историческая отсылка; Cloudflare Pages — текущий хостинг.

## 2026-05-05 — Двухуровневая дизайн-система: primitives с alpha-шкалами + semantic-токены

**Контекст:** По мере роста UI потребовались системные вариации каждого базового цвета (фон карточек, hover, divider, лёгкие подложки). Нужна структура, в которой смена бренда не требует править все компоненты вручную.

**Решение:** Двухуровневая структура в `styles/tokens.css`:
1. **Primitives** — палитра с 9 ступенями alpha (100/80/64/48/32/24/16/8/4) для каждого базового цвета: `--black-*`, `--white-*`, `--light-*`, `--dark-*`, `--red-*`, `--green-*`, `--blue-*`, `--yellow-*`, `--purple-*`, `--toxic-*`. Solid-токен `--brand` без alpha-шкалы.
2. **Semantic** — токены для UI (`--text-primary`, `--surface-bg`, `--border-event`, и т.д.), ссылающиеся на primitives через `var()`.

При изменении бренда меняется только примитив — все семантические токены обновляются автоматически.

**Альтернативы:** Альтернативы не задокументированы в истории.

**Последствия:** Все новые компоненты конструируются из семантических токенов. CSS стал предсказуемее. Параллельно начал расходиться `design-snapshot.md` с реальной структурой `tokens.css` — это привело к Мине 1 в этапе 2 адаптации (2026-05-10) и пересборке снапшота под реальный код.

## 2026-05-06 — Self-host Inter Variable вместо Google Fonts CDN

**Контекст:** Inter — основной шрифт проекта, по умолчанию подключался через Google Fonts CDN. CDN передаёт IP пользователей третьей стороне (Google) — это попадает под определение передачи персональных данных по российскому 152-ФЗ и европейскому GDPR.

**Решение:** Self-host Inter Variable — один variable-файл `/assets/fonts/InterVariable.woff2` (~352 KB) покрывает все веса 100–900. Подключение через `@font-face` в `tokens.css`, без внешних запросов.

**Альтернативы:** Альтернативы не задокументированы в истории.

**Последствия:** Соответствует 152-ФЗ и GDPR без дополнительных решений (явно зафиксировано в комментарии в `styles/tokens.css`). Сайт не зависит от внешнего CDN — стабильнее, без блокировок региональными провайдерами.

## 2026-05-06 — Wildfire 64 как display-шрифт для h1

**Контекст:** Inter не давал нужного «характера» в крупных акцентах (имена в попапе, крупные заголовки в about). Нужен display-шрифт.

**Решение:** Купленный шрифт Wildfire (BBKV) self-hosted в `/assets/fonts/Wildfire.woff2`. Используется только для `--font-h1-*` (имена в попапе, крупные акценты в about). Всё остальное — на Inter.

**Альтернативы:** Альтернативы не задокументированы в истории.

**Последствия:** Покупка лицензии — единственная не-share зависимость в шрифтах проекта. Файл должен лежать строго по пути `/assets/fonts/Wildfire.woff2` — несоответствие пути и фактического расположения было миной (исправлено в этапе 1 адаптации, коммит `80c74cf`).

## 2026-05-08 — Отказ от формы обратной связи (Web3Forms) в пользу mailto + страница политики приватности

**Контекст:** В about-попапе планировалась форма обратной связи. 2026-05-06 (`31c22fd`) подключили внешний сервис Web3Forms для приёма сообщений. Через 5 коммитов и 2 дня (`010f79e`, 2026-05-08) форма была удалена. Это пример «туда-сюда» в истории проекта — добавили внешнюю зависимость и потом отказались.

**Решение:** Контакт автора через `mailto:` (простая HTML-ссылка, без зависимостей). Отдельная страница «Политика конфиденциальности».

**Альтернативы (попробовали и откатили):** Web3Forms — внешний сервис для приёма форм без бэкенда. Подключён 2026-05-06 коммитом `31c22fd`, откачен 2026-05-08 коммитом `010f79e`. Причина отката не зафиксирована в истории отдельно.

**Последствия:** Эпизод стал одним из явных триггеров правила 1.2 в глобальном `CLAUDE.md` («перед добавлением внешней зависимости — задавать уточняющие вопросы, не принимать решения автоматически») — оно прямо ссылается на этот случай как мотивацию.

## 2026-05-10 — Адаптация проекта под глобальную систему правил CLAUDE.md

**Контекст:** Артём составил глобальную «конституцию» работы Claude Code (`~/.claude/CLAUDE.md`) на основе ретроспективы Past Simple — единые правила для всех проектов, плюс шаблоны документации для уровней 1/2/3. Существующий проектный `CLAUDE.md` Past Simple, документация (`SESSION.md`, `BACKLOG.md`, `DEPLOY.md`), `design-snapshot.md`, скиллы и `_schema.js` накопили долг и не соответствовали новому формату. Также накопились stale-данные (диапазон лет, цветовые токены, мёртвые файлы).

**Решение:** Большая разовая работа в 4 этапа в feature-ветке `refactor/adapt-to-global-system`:
- **Этап 1** — переписать проектный `CLAUDE.md` под формат шаблона уровня 2.
- **Этап 2** — починить три «мины» из ретроспективы: `design-snapshot.md` ↔ `tokens.css`, хардкод диапазона лет в скиллах, мёртвые 0-байтные файлы.
- **Этап 3** — переписать `SESSION.md`, `BACKLOG.md`, `DEPLOY.md`; создать `DECISIONS.md` (этот файл) и `README.md`.
- **Этап 4** — обновить `.claude/settings.json` (Stop-hook 30 мин вместо 10), проверить скиллы.

После завершения всех 4 этапов — единый push в `origin/main` и feature-ветка.

**Альтернативы:** Альтернативы не задокументированы в истории.

**Статус:** Завершено 2026-05-10. Все 4 этапа закрыты в feature-ветке `refactor/adapt-to-global-system`.

**Последствия:** 18 коммитов в feature-ветке. Переписан проектный `CLAUDE.md` под формат шаблона уровня 2; починены 3 «мины» из ретроспективы (`design-snapshot.md` ↔ `tokens.css`, хардкод диапазона лет в скиллах и `_schema.js`, мёртвые 0-байтные файлы); 6 файлов документации приведены к стандартному формату (`SESSION.md`, `BACKLOG.md`, `DEPLOY.md`, новые `DECISIONS.md`/`README.md`, `.claude/settings.json`); технический долг по разведке хардкода в `js/` оформлен 3 конкретными записями в `BACKLOG.md` с явными зависимостями. По дороге обнаружены и починены реальные баги (путь к шрифту Wildfire, сломанные пути к `data/` в скиллах, stale-пути в проектном `CLAUDE.md`). `public/data/_schema.js` превращён из файла-комментариев в реальный модуль с экспортами `START_YEAR=1850, END_YEAR=1950, PERSON_CATEGORIES, CONNECTION_TYPES`.
