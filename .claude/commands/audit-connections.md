# /audit-connections

Аудит связей **внутри наших же текстов**. Ищет упоминания других людей из roster в `event.description` и `event.sections[]`, сверяет с `connections[]`. Покрывает R15.

Этот скилл **не дёргает внешние источники** — он работает только с тем, что уже в базе. Дёшев. Запускать перед `/check-person` и `/check-event`, чтобы быстро отловить недостающие связи.

## Аргумент

- `/audit-connections` — по всем людям.
- `/audit-connections <person-id>` — по одному человеку (его events.description + sections, плюс упоминания этого человека в текстах ДРУГИХ).

## Важно: связь визуально работает только если у обоих есть `event` на тот же год

Парный попап (R4), connection-линия на главном экране и точка на линии жизни — все три визуальных эффекта требуют **`event` у обоих людей на один и тот же год**. Просто двусторонняя запись в `connections[]` без событий — данные консистентны, но **визуально ничего не появится**. Поэтому отчёт делит находки на три группы по визуальной применимости.

## Алгоритм

1. **Загрузи список людей** (id + name + fullName + алиасы из bio):
   ```bash
   jq '[.[] | {id, name, fullName}]' public/data/people.json
   ```

2. **Собери словарь имён → id**. Для каждого человека: ключи = `name`, `fullName`, плюс варианты (фамилия отдельно, инициалы). Простой подход: split `fullName` по пробелам, добавь самое длинное слово как алиас (обычно фамилия). Без переусложнения.

3. **Прогон по событиям:**
   - Если задан `<person-id>` — только его `events[]`.
   - Иначе — все `events[]` всех людей.
   - Для каждого события — соедини `description` + все `sections[].body` в один текст.
   - В этом тексте ищи **подстроки** из словаря имён. Используй regex с границами `(?<![А-Яа-яA-Za-z])` и `(?![А-Яа-яA-Za-z])`, чтобы «Дягилев» не матчился внутри другого слова. Падежные формы русского ловятся через корень-подстроку.

4. **Для каждого найденного упоминания** проверь два условия и классифицируй:
   - **C1.** Есть ли уже в `connections[]` у текущего человека запись на упомянутого с этим же годом?
   - **C2.** Есть ли у упомянутого человека event на этот же год?

5. **Классификация находки:**

   | C1 | C2 | Группа | Действие |
   |---|---|---|---|
   | нет | да | **G1: готовые к применению** | Добавить mutual `connection` с указанным годом — визуальный эффект сразу. |
   | нет | нет | **G2: требует нового event у второго** | Решить, нужно ли создать event у второго человека на этот год. Если да — `/add-event` или ручная правка `people.json`. Только после этого добавлять `connection`. |
   | да на другой год | да или нет | **G3: связь есть на другой год** | Решить, добавлять ли отдельную запись `connection` на новый год (для длительных связей одного эпизода — обычно нет; для отдельного значимого события — да). Если C2=нет, кейс автоматически уходит в G2. |

6. **Двусторонняя проверка:**
   - Пройди по `connections[]` каждого человека и убедись, что обратная связь есть. (Дублирует `/check`, но в контексте именно связей.)

7. **Отчёт:**

   ```
   # audit-connections: <person-id или ALL>

   ## G1: готовые к применению (event у обоих, связи нет)

   ### <person-A.id> ↔ <person-B.id> ({year})
   - Контекст: упомянут в <person-A>.events[<year>] секция <idx>
   - Цитата: «...»
   - У <person-B> есть event на <year>: «<title-B>»
   - Предложение: добавить mutual connection (type=?, описание из контекста)

   ## G2: требует нового event у второго (event только у A)

   ### <person-A.id> ↔ <person-B.id> ({year})
   - Контекст: упомянут в <person-A>.events[<year>] секция <idx>
   - Цитата: «...»
   - У <person-B> events[]: ничего на <year> (ближайшие: <list of years>)
   - Действие: решить — создать event у <B> на <year> или оставить связь без визуала. Если создавать — какое событие?

   ## G3: связь уже есть на другой год

   ### <person-A.id> ↔ <person-B.id> ({year события})
   - Связь зафиксирована на годы: [<existing>]
   - Контекст: упомянут в <person-A>.events[<year>]
   - Решение: дубль не нужен (длительная связь) ИЛИ добавить отдельную запись (отдельный эпизод)

   ## Односторонние связи (есть у A, нет у B):
   ...

   ## Итог
   🟢 N в G1 (можно применять), 🟡 M в G2 (нужны новые events), 🔵 K в G3 (решить), ↔ односторонние.
   ```

8. **Применение правок** — по группам:
   - **G1** — самое безопасное, применяй с подтверждением (`all`/`none`/список).
   - **G2** — отдельная задача. Для каждой пары обсудить с пользователем, какое event создать у второго (или не создавать). После создания event — добавление connection делается как в G1.
   - **G3** — индивидуально.
   - `type` связи (`CONNECTION_TYPES` из `_schema.js`) — спросить у пользователя, если из контекста неоднозначно.

9. **После правок** — `/check`.

## Что НЕ делает

- Не дёргает внешние источники. Только наши тексты.
- Не предлагает новых людей в roster — только связи между существующими.
- Не угадывает `type` связи молча.
- **Не применяет G2 правки без явного создания event у второго.** Просто записать mutual `connection` без события — это создаёт «мёртвую» связь без визуала (см. примечание в шапке).
